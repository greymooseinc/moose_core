# Localization (L10n)

> **Current version: 2.3.0**

Runtime string localization for moose_core applications. Plugins register English defaults during startup; `LocalizationPlugin` loads per-locale JSON files on top. The result is a two-layer, hot-switching string system that requires zero code changes to add a new language.

---

## How It Works

```
┌─────────────────────────────────────────────────────┐
│  Layer 2 — Client Overrides (per locale JSON)       │
│  assets/l10n/ja.json, assets/l10n/fr.json, …        │
│  Loaded by LocalizationPlugin after locale switch   │
└──────────────────┬──────────────────────────────────┘
                   │ shadows matching keys
┌──────────────────▼──────────────────────────────────┐
│  Layer 1 — Plugin Defaults (English)                │
│  Registered by each FeaturePlugin.onRegister()      │
│  via appContext.l10n.registerDefaults(pluginId, {}) │
└──────────────────┬──────────────────────────────────┘
                   │ fallback chain
┌──────────────────▼──────────────────────────────────┐
│  Key itself  (prevents blank UI)                    │
└─────────────────────────────────────────────────────┘
```

**Lookup order for `l10n('auth.signIn')`:**

1. Override for active locale (`ja` → `"ログイン"`)
2. English default registered by `AuthPlugin`
3. `fallback` parameter if provided
4. The key string `'auth.signIn'` (developer sees missing key, not blank text)

**Rebuild propagation:**

`MooseL10n` extends `ChangeNotifier`. `MooseScope` listens to it and increments `l10nGeneration` on every `setLocale()` or `applyOverrides()` call. Because `MooseScope` is an `InheritedWidget`, every widget that calls `context.moose` rebuilds automatically — no manual `setState` or `StreamBuilder` needed.

---

## Key Naming Convention

Keys are fully-qualified dot-separated strings: `<pluginId>.<camelCaseKey>`.

```
auth.signIn
cart.emptyMessage
products.noProductsFound
bottomTabs.noTabsConfigured
tabs.home
```

- The plugin-id prefix deduplicates keys across plugins without a central registry.
- The camelCase suffix mirrors the Dart field name convention.
- Keys are case-sensitive.

---

## Template Interpolation

Values may contain `{paramName}` placeholders. Pass replacements via `params`:

```dart
// Registered default: 'Showing {count} results for "{query}"'
context.moose.l10n('search.resultsSummary', params: {
  'count': results.length.toString(),
  'query': searchQuery,
});
// → 'Showing 12 results for "sneakers"'
```

Unmatched placeholders are left as-is — partial substitution is safe.

## MooseL10n API

`MooseL10n` lives at `appContext.l10n` and is exported via `package:moose_core/services.dart`.

### Lookup (widgets / sections)

```dart
// Simple
context.moose.l10n('cart.checkout')

// With params
context.moose.l10n('cart.itemCount', params: {'count': '3'})

// With explicit fallback
context.moose.l10n('myPlugin.title', fallback: 'My Plugin')
```

### Static config-string resolver

```dart
// Resolves value as l10n key if a translation exists; returns value as-is otherwise.
// Use for config strings from environment.json that may be either literals or keys.
MooseL10n.resolve('tabs.home', context.moose.l10n)   // → 'Home' / 'ホーム'
MooseL10n.resolve('MY STORE', context.moose.l10n)    // → 'MY STORE' (no key match)
```

This is how `MooseAppBar` resolves its `title` setting and `BottomTabsScreen` resolves tab labels — the caller does not need to know whether the string is a literal or a key.

### Plugin registration

```dart
// In FeaturePlugin.onRegister():
appContext.l10n.registerDefaults('myPlugin', {
  'title':         'My Plugin',
  'emptyMessage':  'Nothing here yet',
  'itemCount':     '{count} items',   // supports placeholders
});
```

### Locale management (LocalizationPlugin only)

```dart
appContext.l10n.setLocale('ja');                    // switch active locale
appContext.l10n.applyOverrides('ja', {'auth.signIn': 'ログイン'});
appContext.l10n.activeLocale                        // → 'ja'
appContext.l10n.hasOverridesFor('ja')               // → true
appContext.l10n.generation                          // monotonic counter; changes on every setLocale/applyOverrides
```

### Hooks (for cross-plugin translation)

Registered by `LocalizationPlugin` during `onInit()`:

```dart
// Translate a key from another plugin
final text = hookRegistry.execute<String>('l10n:translate', {
  'key': 'cart.checkout',
  'fallback': 'Checkout',
  'params': {'count': '3'},
});

// Get active locale code
final locale = hookRegistry.execute<String>('l10n:get_locale', null);
// → 'ja'
```

## Plugin Developer Guide

### 1. Register defaults in `onRegister()`

```dart
@override
void onRegister() {
  appContext.l10n.registerDefaults('loyalty', {
    'title':              'Loyalty Points',
    'pointsBalance':      'You have {points} points',
    'redeemButton':       'Redeem Points',
    'insufficientPoints': 'Not enough points',
    'earnedMessage':      'You earned {points} points on this order',
  });

  // ... rest of registration
}
```

**Rules:**
- Call `registerDefaults()` at the top of `onRegister()`, before any other registration.
- Always use English values — these are the baseline.
- Key suffix is bare camelCase (no plugin prefix); the prefix is added automatically.
- Multiple calls for the same `pluginId` accumulate; later values overwrite duplicates.

### 2. Use l10n in sections

```dart
@override
Widget build(BuildContext context) {
  final l10n = context.moose.l10n;

  return Column(
    children: [
      Text(l10n('loyalty.title')),
      Text(l10n('loyalty.pointsBalance', params: {'points': points.toString()})),
      ElevatedButton(
        onPressed: _redeem,
        child: Text(l10n('loyalty.redeemButton')),
      ),
    ],
  );
}
```

### 3. Use l10n in BLoCs (avoid — prefer passing resolved strings from widgets)

BLoCs have no `BuildContext`. If a BLoC must format an error message:

```dart
// Inject the l10n instance via constructor
class LoyaltyBloc extends Bloc<LoyaltyEvent, LoyaltyState> {
  LoyaltyBloc({required this.repository, required this.l10n}) : super(...);
  final MooseL10n l10n;
}

// Pass it from the plugin
LoyaltyBloc(repository: repo, l10n: appContext.l10n);
```

Or pass the resolved string in the event rather than resolving inside the BLoC.

### 4. Config strings as l10n keys

Any setting in `environment.json` that is a display string can use an l10n key as its value. No special syntax — resolution is transparent:

```json
{
  "name": "loyalty.section.banner",
  "settings": {
    "title": "loyalty.bannerTitle",
    "emptyMessage": "loyalty.noBonusesAvailable"
  }
}
```

```dart
// In the section build():
final title = MooseL10n.resolve(getSetting<String>('title'), context.moose.l10n);
```

If `loyalty.bannerTitle` is registered, it resolves to the translated string. If not registered, `getSetting<String>('title')` returns `'loyalty.bannerTitle'` — which `resolve()` also returns unchanged, so the key name appears in the UI (useful for debugging).

## Adding a New Language

### Step 1 — Create the override file

```
assets/l10n/fr.json
```

```json
{
  "auth.signIn":                "Se connecter",
  "auth.signOut":               "Se déconnecter",
  "cart.checkout":              "Passer à la caisse",
  "cart.emptyMessage":          "Votre panier est vide",
  "products.noProductsFound":   "Aucun produit trouvé",
  "products.minimumRatingChip": "⭐ {rating}+",
  "tabs.home":                  "Accueil",
  "tabs.shop":                  "Boutique"
}
```

**Format rules:**
- Flat map — no nesting.
- Keys are fully-qualified (`pluginId.camelCaseKey`).
- Only include keys that differ from the English default. Missing keys fall back to English silently.
- Preserve `{placeholder}` markers exactly as they appear in the English default.

### Step 2 — Register the asset in `pubspec.yaml`

```yaml
flutter:
  assets:
    - assets/l10n/
```

A glob covers all files in the folder; no per-file entries needed.

### Step 3 — Add the locale to `LocaleManagerPlugin` settings

```json
{
  "id": "locale_manager",
  "settings": {
    "defaultLocale": "en",
    "supportedLocales": ["en", "fr"]
  }
}
```

That's it. `LocalizationPlugin` loads the file automatically when the locale switches.

## LocalizationPlugin

**Class:** `LocalizationPlugin extends FeaturePlugin`
**Location:** `lib/plugins/localization/localization_plugin.dart`

Bridges `LocaleManagerPlugin` (which owns `MaterialApp.locale`) with `MooseL10n` (which serves strings to widgets). It must be registered **after** `LocaleManagerPlugin`:

```dart
plugins: [
  () => LocaleManagerPlugin(),
  () => LocalizationPlugin(),   // ← after locale manager
  // ... all other plugins
],
```

**Startup sequence:**

1. `onInit()` fires after all plugins have registered their defaults.
2. Reads the initial locale from `LocaleManagerPlugin` via `moose.locale.hook.get_locale`.
3. Calls `appContext.l10n.setLocale(initialLocale)`.
4. Loads `assets/l10n/<initialLocale>.json` and calls `applyOverrides()`.
5. Subscribes to `moose.locale.event.changed` for future switches.

**On locale switch:**

```
moose.locale.event.changed
  → setLocale(locale)       — widgets with registered defaults rebuild immediately
  → _loadOverrides(locale)  — loads JSON file (async)
  → applyOverrides()        — second rebuild with translated strings
```

Two rebuilds occur intentionally: the first is instant (clears stale locale strings), the second arrives when the JSON finishes loading (populates translated strings). This avoids a flash of the previous locale.

**English apps:** No JSON file needed — all strings come from plugin defaults. `LocalizationPlugin` can still be registered (it's a no-op for `'en'`).

## Rebuild Mechanism

Understanding how locale changes propagate to widgets:

```
LocalizationPlugin
  → appContext.l10n.setLocale('ja')
    → _generation++
    → notifyListeners()
      → MooseScope._onL10nChanged()
        → setState(() {})
          → MooseScope.build() passes new l10nGeneration to _MooseScopeInherited
            → _MooseScopeInherited.updateShouldNotify() returns true
              → all context.moose dependants rebuild
```

`appContext.l10n.applyOverrides()` follows the same path when the JSON file finishes loading — triggering the second rebuild.

**Key property:** `MooseL10n.generation` is a monotonically increasing `int`. `MooseScope` uses it (not the locale string) for change detection, so two `applyOverrides()` calls for the same locale both trigger rebuilds correctly.

## Common Pitfalls

**Registering defaults in `onInit()` instead of `onRegister()`**

```dart
// ❌ Wrong — onInit() is for async I/O; other plugins may try to resolve
//    your strings during their own onRegister() and get blank keys.
@override
Future<void> onInit() async {
  appContext.l10n.registerDefaults('myPlugin', {...});
}

// ✅ Correct
@override
void onRegister() {
  appContext.l10n.registerDefaults('myPlugin', {...});
}
```

**Using a plugin-prefixed key in `registerDefaults()`**

```dart
// ❌ Wrong — prefix is added automatically; this registers 'myPlugin.myPlugin.title'
appContext.l10n.registerDefaults('myPlugin', {
  'myPlugin.title': 'My Plugin',
});

// ✅ Correct — bare camelCase suffix only
appContext.l10n.registerDefaults('myPlugin', {
  'title': 'My Plugin',
});
```

**Nesting keys in the JSON override file**

```json
// ❌ Wrong — MooseL10n expects a flat map
{
  "auth": {
    "signIn": "Se connecter"
  }
}

// ✅ Correct — flat, fully-qualified keys
{
  "auth.signIn": "Se connecter"
}
```

**Missing `{placeholder}` in the translated string**

If the English default is `'You have {points} points'` and the Japanese version omits the placeholder:

```json
// ❌ Wrong — {points} will not be substituted
{ "loyalty.pointsBalance": "ポイント残高" }

// ✅ Correct — keep the placeholder
{ "loyalty.pointsBalance": "{points}ポイント" }
```

**Forgetting `LocalizationPlugin` before other plugins**

`LocalizationPlugin` must come after `LocaleManagerPlugin` but can be in any position relative to feature plugins, because `registerDefaults()` is always called during `onRegister()` (before `onInit()` where `LocalizationPlugin` does its work).

```dart
plugins: [
  () => LocaleManagerPlugin(),
  () => LocalizationPlugin(),  // ← before feature plugins that call l10n in onInit
  () => CartPlugin(),
  () => ProductsPlugin(),
],
```

**Using `l10n` in a `FeaturePlugin.onRegister()` before overrides are loaded**

Override files are loaded in `LocalizationPlugin.onInit()`. Calling `appContext.l10n('myPlugin.foo')` during another plugin's `onRegister()` returns the English default — which is correct for English but won't include overrides yet. This is expected and not a bug. Widgets are built after all plugins initialize, so users always see the correct locale.

## File Reference

| File | Role |
|---|---|
| `moose_core/lib/src/l10n/moose_l10n.dart` | Core `MooseL10n` class — defaults, overrides, lookup, `resolve()` |
| `moose_core/lib/src/app/moose_app_context.dart` | Owns `final MooseL10n l10n` field |
| `moose_core/lib/src/app/moose_scope.dart` | Listens to `l10n` and propagates generation changes to widget tree |
| `moose_core/lib/src/ui/moose_app_bar.dart` | Uses `MooseL10n.resolve()` for the `title` setting |
| `lib/plugins/localization/localization_plugin.dart` | Loads JSON overrides; bridges EventBus locale events to `MooseL10n` |
| `lib/plugins/locale_manager/locale_manager_plugin.dart` | Owns `MaterialApp.locale`; fires `moose.locale.event.changed` |
| `lib/plugins/bottom_tabs/bottom_tabs_plugin.dart` | Registers `tabs.*` defaults; `BottomTabsScreen` uses `MooseL10n.resolve()` for labels |
| `assets/l10n/*.json` | Per-locale override files (one per supported language) |
